# Flow AST

The Flow AST is a strongly-typed BPMN 2.0.2 graph model where all element connections use object references rather than string IDs, and event semantics are decomposed into orthogonal *position* x *definition* axes. The model lives entirely in `Schemata.Flow.Skeleton/Models/` and is consumed by both the DSL builders and the engine at runtime.

## Where the code lives

| Package | Key files |
| --- | --- |
| `Schemata.Flow.Skeleton` | `Models/FlowElement.cs`, `Models/Activity.cs`, `Models/Gateway.cs`, `Models/FlowEvent.cs`, `Models/SequenceFlow.cs`, `Models/ProcessDefinition.cs`, `Models/ProcessInstance.cs`, `Models/IEventDefinition.cs` |

## FlowElement

All graph nodes inherit from `FlowElement`:

```csharp
public abstract class FlowElement
{
    public string Id   { get; set; }
    public string Name { get; set; }
}
```

`Name` doubles as the process instance state label during execution. `Id` is a GUID string generated by `ProcessDefinition.GenerateId()` during magic-property initialization.

```
        EmbeddedSubProcess

        AdHocSubProcess

```

## Gateway hierarchy

```
Gateway : FlowElement
    ExclusiveGateway
    ParallelGateway
    InclusiveGateway
    EventBasedGateway
        Parallel : bool          // false = exclusive (default), true = parallel
    ComplexGateway
```

The `StateMachineEngine` supports only `ExclusiveGateway` and `EventBasedGateway`. Attempting to traverse a `ParallelGateway` or `InclusiveGateway` throws `FailedPreconditionException`.

## FlowEvent

```csharp
public class FlowEvent : FlowElement
{
    public EventPosition      Position    { get; set; }
    public IEventDefinition?  Definition  { get; set; }
    public bool               Interrupting { get; set; } = true;
    public Activity?          AttachedTo  { get; set; }
    public bool               IsTerminate { get; set; }
}
```

`StartEvent` and `EndEvent` are thin subclasses whose constructors set `Position` to the appropriate value. They exist to support the magic-property discovery pattern: declaring `public EndEvent Foo { get; private set; }` on a `ProcessDefinition` subclass causes the base constructor to create a `FlowEvent` with `Position = End`.

### EventPosition

```csharp
public enum EventPosition
{
    Start,              // creates a new token
    IntermediateCatch,  // waits for a trigger
    IntermediateThrow,  // fires when token arrives
    Boundary,           // attached to an Activity
    End,                // consumes the token
}
```

## IEventDefinition hierarchy

```
IEventDefinition
    Name : string                   // used for trigger matching at runtime

    NoneDefinition
    Message  { PayloadType? : Type }
    Signal   { PayloadType? : Type }
    TimerDefinition
        TimerType       : Date | Duration | Cycle
        TimeExpression  : string    // ISO 8601 or cron
    ErrorDefinition
        ErrorCode?      : string
        ExceptionType   : Type
    EscalationDefinition { EscalationCode? : string }
    ConditionalDefinition { Condition : IConditionExpression }
    CompensationDefinition { ActivityRef? : Activity }
    CancelDefinition
    LinkDefinition
    MultipleDefinition  { Definitions : List<IEventDefinition> }
    ParallelDefinition  { Definitions : List<IEventDefinition> }
```

Event definitions are reusable: the same `Message Pay` instance can appear in both `Start(Pay)` and `On(Pay)`. Position is determined by the DSL usage context, not by the definition type.

## SequenceFlow

```csharp
public sealed class SequenceFlow
{
    public string                Id        { get; set; }
    public FlowElement           Source    { get; set; }
    public FlowElement           Target    { get; set; }
    public IConditionExpression? Condition { get; set; }
    public bool                  IsDefault { get; set; }
}
```

`Source` and `Target` hold direct object references, not string IDs. This enables reference-based matching during engine traversal and eliminates dangling string references.

## IConditionExpression

```csharp
public interface IConditionExpression
{
    ValueTask<bool> Evaluate(FlowConditionContext context);
}
```

`LambdaConditionExpression` wraps a `Func<FlowConditionContext, ValueTask<bool>>`. For typed conditions (`When<T>(...)`), the lambda extracts the variable of type `T` from `Variables` by the parameter name inferred from the expression, deserializes from JSON, and evaluates the compiled predicate.

## ProcessDefinition

```csharp
public class ProcessDefinition
{
    public string Name         { get; set; }
    public string? DisplayName { get; set; }
    public string? Description { get; set; }

    public List<FlowElement>          Elements    { get; }
    public List<SequenceFlow>         Flows       { get; }
    public List<Message>              Messages    { get; }
    public List<Signal>               Signals     { get; }
    public List<ErrorDefinition>      Errors      { get; }
    public List<EscalationDefinition> Escalations { get; }
}
```

The base constructor calls `InitializeProperties()`, which scans the concrete subclass for **magic properties**: public auto-properties whose value is `null` and whose type is one of the known declarable types (`Activity` subclasses, `StartEvent`, `EndEvent`, `FlowEvent`, `Message`, `Signal`, `ErrorDefinition`, `EscalationDefinition`). For each, it creates an instance and writes it back via the compiler-generated backing field. `[DisplayName("...")]` on the property overrides `Name` with the attribute value.

## ProcessInstance

```csharp
public sealed class ProcessInstance
{
    public string?                      StateId     { get; set; }
    public string?                      State       { get; set; }
    public string?                      WaitingAtId { get; set; }
    public string?                      WaitingAt   { get; set; }
    public bool                         IsComplete  { get; set; }
    public Dictionary<string, object?>  Variables   { get; set; }
}
```

`WaitingAt` distinguishes "Activity executing" from "waiting for event trigger":

- When `WaitingAt` is set, `AdvanceAsync` returns early.
- `TriggerAsync` checks `WaitingAt` first when resolving event matches.
- API consumers can query this to determine if the instance needs input.

## Extension points

- Implement `IConditionExpression` to plug in a custom expression evaluator (e.g., CEL or AIP-160).
- Implement `IFlowRuntime` to support a different execution model (parallel gateways, compensation).
- Implement `IFlowEngineValidator` to add engine-specific validation rules.

## Design motivation

Using object references instead of string IDs in `SequenceFlow` eliminates an entire class of runtime errors (dangling references, case-sensitivity bugs). The position x definition decomposition for events mirrors the BPMN 2.0.2 spec and allows the same definition object to be reused across multiple positions without duplication.

## Caveats

- `ProcessDefinition.InitializeProperties()` uses reflection to write backing fields. Properties must be auto-properties (compiler-generated `<Name>k__BackingField`) for this to work. Hand-written backing fields are not discovered.
- `TimerDefinition.TimeExpression` is a raw string. The `StateMachineEngine` does not parse it; `Schemata.Flow.Scheduling` converts it via `TimerDefinitionConverter.ToSchedule`.
