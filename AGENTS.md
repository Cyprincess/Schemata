# Agent Instructions

.NET application framework for modular, extensible business applications.

## Target Frameworks

- **Runtime packages** (`src/`): `net8.0;net10.0` — dual targeting
- **Source generators** (`generators/`): `netstandard2.0`
- **Tests** (`tests/`): `net8.0;net10.0`
- **Meta packages** (`targets/`): `net8.0;net10.0`

The SDK is pinned to `10.0.201` in `global.json` with `allowPrerelease: true` and `rollForward: major`.

## Package Layer Convention

Every feature domain follows a three-layer pattern:

| Layer | Suffix | Purpose |
|---|---|---|
| Contracts | `*.Skeleton` | Interfaces, base classes, abstract types |
| Implementation | `*.Foundation` | Concrete logic + feature registration (`ISimpleFeature`) |
| Provider | `*.Redis`, `*.RabbitMq`, etc. | Specific technology adapter |

The Foundation layer depends on its own Skeleton. Provider packages depend on Skeleton only.

## Feature Architecture

Features implement `ISimpleFeature` (or extend `FeatureBase`). They register via `SchemataBuilder.AddFeature<T>()` and participate in three lifecycle phases:

- `ConfigureServices(…)` — service registration (ordered by `Order`)
- `ConfigureApplication(…)` — middleware pipeline (ordered by `Priority`)
- `ConfigureEndpoints(…)` — endpoint mapping (ordered by `Priority`)

The range `[100_000_000, 900_000_000]` is reserved. Extension features (from other packages) use `400_000_000+`. See the README feature tables for exact values — adding a feature between two existing ones requires checking both tables.

Built-in features in `src/Schemata.Core/Features/` are activated via `UseXxx()` extension methods on `SchemataBuilder`. See `SchemataBuilder.cs` and the feature ordering table in README.

## Code Conventions

- **Nullable enabled** globally, **implicit usings disabled**
- **LangVersion: preview**, analysis level: preview
- **File-scoped namespaces** (`namespace Foo.Bar;` — semicolon, no braces)
- Every `src/` project includes a `FodyWeavers.xml` with `ConfigureAwait ContinueOnCapturedContext="false"` and a source generator reference (`Schemata.Advice.Generator` as analyzer)
- Strong-named assemblies in `src/` and `targets/`; not in tests

## Testing

- **Framework**: xUnit (VSTest runner)
- **Mocking**: Moq
- **Coverage**: coverlet.collector with opencover format
- **Naming**: Test classes use `XxxShould` suffix (e.g., `ConfiguratorsShould.cs`, `TokenHandlerShould.cs`)
- **Method names**: `PascalCase_Underscore` (e.g., `Set_Get_RoundTrip`, `TryGet_Missing_ReturnFalse`)
- Unit tests go in `tests/Schemata.Xxx.Tests/`, integration tests in `tests/Schemata.Xxx.Integration.Tests/`

## Dependencies & Tooling

- **Versioning**: `eng/Versions.props` — version is `10.0.0-preview`. Bump here for releases.
- **Central Package Management**: `Directory.Packages.props` — all NuGet versions. TF-conditional versions exist for net8.0 vs net10.0.
- **Conventional commits**: Enforced by cocogitto (`cog.toml`). The `chore` type omits from changelog.
- **Changelog**: Generated by cocogitto into `docs/CHANGELOG.md` using `docs/CHANGELOG.tera`.
- **Documentation**: DocFX (`docs/docfx.json`), deployed to GitHub Pages via `docs.yml` workflow.
- **Source submodule**: `specs/cel` — Google CEL specification. Submodule recursive clone required.

## Package Feeds

Two additional NuGet feeds beyond nuget.org:
- `dotnet-eng`: `https://pkgs.dev.azure.com/dnceng/public/_packaging/dotnet-eng/nuget/v3/index.json`
- `dotnet-tools`: `https://pkgs.dev.azure.com/dnceng/public/_packaging/dotnet-tools/nuget/v3/index.json`

Both are needed for Arcade SDK dependencies.

## CI

- `build.yml`: Matrix build (win/mac/linux) via Arcade CIBuild. Pushes NuGet packages from `master`/tags to NuGet.org.
- `analysis.yml`: SonarCloud analysis + cocogitto conventional commit check.
- `docs.yml`: DocFX site generation + GitHub Pages deploy.

All CI workflows require `submodules: recursive` and cache `.dotnet` + `.packages`.
